Deepgram Flux. Over a raw WebSocket.

Your mic streams 16-bit PCM at 16 kHz in 100 ms chunks. Flux streams back structured events. When it emits an EndOfTurn, we fire the transcript into the agent.

data flow
Microphone
decibri · native WASAPI/CoreAudio
16-bit PCM · 16kHz
100ms chunks, streamed
Deepgram Flux
wss://api.deepgram.com/v2/listen
TurnInfo · EndOfTurn
built-in end-of-turn detection
Agent receives transcript
fire onTranscript(evt.transcript)
  • Flux has end-of-turn detection built in. Unlike traditional STT, Flux knows when you're done speaking. No endpointing timer to tune.
  • Not wired through OpenClaw's STT slot. OpenClaw's STT was designed for telephony. 110 lines of WebSocket is the right fit for a local CLI mic.
  • One API key. No SDK. No OAuth. A single DEEPGRAM_API_KEY and a WebSocket URL. That's the whole integration.
  • Tune with eot_threshold. Default works well for short food-ordering sentences. Raise it for slower speakers.
honesty check · out of scope

What this tutorial won't cover.

Four things are deliberately out of scope so we can ship a clean, running agent.

Wake word detection

The mic is always on while the agent isn't speaking. No "Hey Claw" trigger.

Multi-turn memory across sessions

Every restart starts fresh. Session key is per-process.

Order cancellation

Swiggy's MCP doesn't expose it. The skill routes the user to customer care.

Production hardening

Single-user CLI. No auth, no rate limits, no observability. Don't ship it as-is.

One latency note. Every sentence plays as soon as the model emits it — that's the streaming pipeline. But tool chains still take as long as tool chains take. Filler audio covers the wait. Don't expect sub-second responses on tool-heavy turns.

swiggy oauth · one-time link

One command links your Swiggy.

Run mcporter auth swiggy-food. Browser opens, you log in, you approve. Token's saved locally. Make sure you have at least one saved delivery address in the Swiggy app — the agent uses that, not live GPS.

what happens
  • Browser opens to Swiggy login
  • Log in with your normal Swiggy credentials
  • Approve the MCP access scope
  • Token saved locally — you're linked
why it works this way

OpenClaw's Swiggy skill calls Swiggy's MCP server through mcporter, which holds the OAuth token for you. No manual token shuffling, no copy-pasting credentials — same login flow you already use in the Swiggy app.

Windows gotcha. mcporter truncates the OAuth URL in some shells. If the browser shows client_id and redirect_uri are required, rerun with mcporter --log-level debug auth swiggy-food --reset and paste the full URL from the debug log manually.
murf falcon · tts plugin

Giving the agent its voice.

OpenClaw handles TTS through a standard plugin system. We drop in Murf Falcon — engineered for low latency, ~130 ms time-to-first-audio. One plugin entry, one voice config, one key. Then OpenClaw's streaming hooks do the rest.

  • Why auto: off? We don't want one shot. We want every sentence played as it arrives. Turning auto off hands control of synthesis back to us.
  • OpenClaw exposes block + token hooks. Same pipeline as Matrix, WhatsApp, Telegram. Subscribe to both — per-block synthesis for clean sentences, per-token fallback for tool-heavy batched replies.
  • ~30 lines of core streaming logic. Each block calls synthesize() in parallel. A promise chain serializes playback so chunk two plays before chunk three.
  • Don't double-load. Enabled in openclaw.json + manually imported in code = OOM on second startup. Pick one.

The streaming TTS pipeline

LLM block → onBlockReply → Murf.synthesize() → queued playback → speakers

One install. The agent can order food.

Skills are how OpenClaw agents learn to do things. The Swiggy skill lives on ClawHub — OpenClaw's package registry. Install it, list it as active in openclaw.json, and the LLM figures out the rest by reading a markdown file.

the voice loop · four files

Four files. This is the code you write.

Everything up to now was config. This is the orchestration — the glue that turns install-and-config into a working voice agent.

ear.ts~110 LOC

Mic capture + Deepgram WS

decibri captures 16-bit PCM at 16 kHz in 100 ms chunks. Each chunk pushes into a Flux WebSocket. On EndOfTurn with a non-empty transcript, fire the callback.

brain.ts~500 LOC

Streaming TTS pipeline · headline

Subscribes to OpenClaw's onBlockReply — every block goes to Murf in parallel. A promise chain serializes playback so chunk two plays before chunk three. onPartialReply is the token-level fallback.

voice.ts~140 LOC

Speaker output

Two channels. playAudio for one-shot (fillers, intros). createPlaybackStream for the streaming queue. stopOneShotPlayback kills an in-progress filler when the first real chunk arrives.

index.ts~140 LOC

The event loop · orchestrator

Startup: render banner, kick off OpenClaw warmup in background, play intro, open mic. On transcript: stop mic, play filler, call chat(), reopen mic when the last chunk finishes. That's a turn.

latency · feeling conversational

Four things make it feel conversational.

Murf stays fast throughout. The latency comes from elsewhere — cold starts, tool chains, waiting for the LLM. These four moves hide all of it.

01 · HEADLINE

Streaming TTS

The big one. OpenClaw's block streaming hooks let us synthesize and play each sentence as it arrives. User hears the first sentence within a couple of seconds of the model starting.

02 · BUILD-TIME

Pre-baked filler WAVs

A script hits Murf once at build time to synth "One moment please" and "I'm checking on that". Saved to assets/. Runtime plays from disk — zero API latency.

03 · TIMING TRICK

Filler before the LLM

The moment a transcript finalizes, fire the filler. User hears "One moment" within 100 ms. Meanwhile the LLM and tool chain grind in the background.

04 · STARTUP

Background warmup

setImmediate(() => warmup()) at startup. Runs while the user is hearing the intro. Amortizes cold-start into seconds the user is already waiting through.
